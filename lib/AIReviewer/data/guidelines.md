## 1. Design

### 1.1 App Name
- Use the brand name for your app. Company names are not allowed.
- May not use the trademarks "Homey" or "Athom" in the app name.
- May not include protocol names (Zigbee, Z-Wave, 433 MHz, Infrared, BLE, Thread or Matter) in the app name.
- Names longer than 4 words are not allowed.

### 1.2 Description
- Using the app's name or repeating text from the readme in the description is not allowed.
- Description should be an engaging one-liner that highlights the purpose of the app.
- Avoid descriptions that simply state support is being added (e.g. "Adds support for Sonos").
- Consider using brand slogans as descriptions.

### 1.3 Readme
- Keep the text short and concise — one to two paragraphs maximum.
- Do not list all the different features, capabilities, and Flow cards available.
- Markdown format in the readme is not allowed and will not be rendered.
- URLs in the readme are not allowed. This includes donation/sponsorship links (PayPal, Ko-fi, Buy Me a Coffee, GitHub Sponsors, etc.); use the `contributing.donate` field in `app.json` instead.
- Don't create a changelog in the readme.

### 1.4 Images
- All images must be provided in either `.jpg` or `.png` format.
- App images must be provided in two resolutions: 250×175 and 500×350 (1000×700 optional).
- Driver images must be provided in two resolutions: 75×75 and 500×500 (1000×1000 optional).
- Images with big two-dimensional unicolored shapes on a monochrome or transparent background are not approved.
- Avoid a logo as the app image.
- Avoid clipart-type images.
- Avoid images that contain Android or iOS app examples.

### 1.4.3 Driver Images
- Provide individual images for each driver.
- A driver image should have a white background and a recognizable picture of the device.
- Don't use the app image as a driver image.
- Don't use the app icon as a driver image.

### 1.5 App Icon
- Should be recognisable at small sizes, and have a transparent background.
- Do not use images, filled illustrations, gradients, or background colours.
- If the app supports a specific brand, use the company's brand icon.
- Icons must have a transparent background.
- Always use the full canvas (960×960px).
- Don't use a driver icon as the app icon.
- Don't use a background color in the icon.
- The default Homey rocket icon (the placeholder shipped by `homey app create`) is not allowed as the final app icon.

### 1.6 Driver Icons
- Canvas size should be 960×960px.
- Always use the full canvas so the icon is displayed properly.
- Where possible, use an angle from the right side rather than a front-facing view.
- Icons must have a transparent background.
- Do not re-use the app icon for drivers.
- Use the correct line width.
- Don't use driver images as an icon.
- Don't use a background color — icons should have a transparent background.

### 1.7 Brand Color
- The `brandColor` property is mandatory in the App Manifest.
- Both the app icon and driver icons must be clearly visible against the brand color.

### 1.8 URLs
- Offer extra help or information by adding URLs (support, source, homepage) to the App Manifest.

### 1.9 Flow Cards
- Flow card titles need to be short and clear.
- Device names should not be mentioned in the title.
- Don't add the When/And/Then statements to the title.
- Do not use parentheses in Flow titles.
- For formatted titles: the title must still be straightforward and clear with the arguments incorporated.
- In case the function of the Flow card is not obvious, use the `hint` property.

### 1.10 Widget Previews
- Both light and dark mode versions must be provided.
- Don't include text in previews.
- Use simple shapes.
- Don't use the same colors as the Widget picker background.
- Try not to use too many different colors.
- Previews should have a transparent background.

### 1.11 Language & Translations
- English is the required language.
- The app must not have any typos, language, or spelling errors. Apps will be rejected if spelling errors are found.
- A partially translated app is confusing for users — avoid sporadic translations throughout the app.
- Maintain consistency: if the description is translated to a language, the readme should also be translated in that language.

### 1.12 Dependencies
- Not permitted to make one app fully dependent on another app.
- The app's core functionality must always work standalone.
- Not allowed to publish an app which does not add any value by itself.

### 1.13 Account
- A developer account name cannot contain emojis, special characters, or inappropriate language.
- For verified developers: the account name should be the name of the company publishing the app.

### 1.14 SDK Version
- New apps must be built on Homey Apps SDK v3. Submissions on older SDK versions will be rejected.
- Existing apps already live on an older SDK may keep their SDK, but new development should target v3.

### 1.15 Permissions
- Permissions in `app.json` must be justified by the app's core functionality. Do not request more than the app actually needs.
- The `manager:homey:api` permission is a broad grant and is only appropriate for apps whose primary purpose requires programmatic access to Homey (e.g. tooling, automations that inspect Homey state). It is not needed to control devices via drivers, and requests that are not clearly justified will be rejected.

## 2. Legal

### 2.1.1 Duplicate Apps
- Ideally there is only one app per brand or concept in the App Store.
- Developers should check the App Store for similar existing apps.
- If a similar app exists, reach out to the existing developer first and explore collaboration.
- Submissions that resemble an existing app without any clarification will be rejected.

### 2.1.2 Code
- Copying or using code created by fellow developers without consent is an infringement.
- Always ask for permission and give credit to the original source in the app manifest.
- If it is revealed that code has been used without consent, the app is at risk of removal.

### 2.2 Explicit Content
- Apps with adult content (e.g. pornography) are not allowed.

### 2.3 Compensation
- Apps that require payment for partial or all functionality are not allowed.
- Apps should be free of charge for all users.
- Exception: apps connecting to paid external services may be free in the store (where payment occurs externally).

## 3. After Submission

### 3.1 Review Duration
- The review process can take up to 2 weeks.

### 3.2 Testing (Verified Developers)
- Provide a few sample devices prior to submission.
- Apps should be thoroughly tested before submission.
- Testing requirements include: device controls, advanced settings, Flow creation, manual state verification, pairing instructions, error messages, Flow card titles, label/hint texts, and translations.

### 3.3 Feedback
- Any findings or feedback during the review process will be shared with the developer and are expected to be implemented.

### 3.4 Removal
- Apps may be hidden or removed if they are no longer compatible with Homey, functioning correctly, or actively maintained.
- Apps without an update in two years or more may be considered abandoned.
